
### 日志
理解 LevelDB 的日志系统（Write-Ahead Log, WAL），建议按以下顺序阅读相关代码：

1.  **日志记录格式 (Log Record Format)**:
    *   首先理解 WAL 文件中每条记录是如何组织的。日志记录可能会被切分成多个物理的块 (chunk)。
    *   [`db/log_format.h`](db/log_format.h ): 这个头文件定义了日志块的大小 (`kBlockSize`)、记录类型 (`RecordType`: `kZeroType`, `kFullType`, `kFirstType`, `kMiddleType`, `kLastType`) 以及每个块头部的格式（校验和、长度、类型）。

2.  **日志写入器 (Log Writer)**:
    *   了解数据是如何被格式化并写入到日志文件中的。
    *   [`db/log_writer.h`](db/log_writer.h ) 和 [`db/log_writer.cc`](db/log_writer.cc ):
        *   `log::Writer` 类负责将用户数据（通常是序列化后的 `WriteBatch`）封装成一条或多条日志记录，并写入到磁盘文件中。
        *   `Writer::AddRecord()`: 核心方法，接收一个 `Slice` 作为用户数据，将其切分成合适的块，计算校验和，然后写入文件。它会处理记录跨越物理块边界的情况。

3.  **日志读取器 (Log Reader)**:
    *   了解如何从日志文件中读取并解析记录，这在数据库恢复时至关重要。
    *   [`db/log_reader.h`](db/log_reader.h ) 和 [`db/log_reader.cc`](db/log_reader.cc ):
        *   `log::Reader` 类负责从日志文件中读取物理块，并从中重组出完整的日志记录。
        *   `Reader::ReadRecord()`: 核心方法，尝试读取并返回下一条完整的日志记录。它会处理校验和验证、记录跨块等情况。

4.  **WriteBatch 的序列化**:
    *   WAL 中存储的是 `WriteBatch` 的内容。
    *   [`include/leveldb/write_batch.h`](include/leveldb/write_batch.h ) 和 [`db/write_batch.cc`](db/write_batch.cc ):
        *   `WriteBatchInternal::Contents()`: 返回 `WriteBatch` 内部的序列化表示（一个 `Slice`），这个 `Slice` 就是传递给 `log::Writer::AddRecord()` 的内容。
        *   `WriteBatchInternal::SetContents()`: 用于从 `Slice` 反序列化恢复 `WriteBatch`。

5.  **数据库操作与日志的集成 - 写入流程**:
    *   了解在执行写操作（Put/Delete）时，日志是如何被写入的。
    *   [`db/db_impl.cc`](db/db_impl.cc ):
        *   `DBImpl::Write()`: 这个方法是处理写请求的核心。在将数据写入 MemTable 之前（或在构建 `WriteBatch` 组之后），它会将 `WriteBatch` 的内容通过 `log_->AddRecord()` 写入到当前的 WAL 文件中。
        *   `DBImpl` 成员 `log_` (类型为 `log::Writer*`) 指向当前的日志写入器。
        *   `DBImpl` 成员 `logfile_` (类型为 `WritableFile*`) 指向当前的日志文件。
    *   [`include/leveldb/options.h`](include/leveldb/options.h ): `WriteOptions::sync` 成员决定了每次写日志后是否需要执行 `fsync` 来确保数据落盘。

6.  **数据库操作与日志的集成 - 恢复流程**:
    *   了解数据库在启动时如何使用日志文件进行数据恢复。
    *   [`db/db_impl.cc`](db/db_impl.cc ):
        *   `DBImpl::Recover()`: 当数据库打开时，如果存在未被处理的日志文件，这个方法会被调用。它会创建一个 `log::Reader` 来读取日志文件中的记录。
        *   对于每条成功读取的记录（代表一个 `WriteBatch`），它会使用 `WriteBatchInternal::InsertInto()` 将 `WriteBatch` 中的操作应用到 MemTable 中，从而恢复数据库到崩溃前的状态。
        *   `DBImpl::Open()` 方法会调用 `Recover()`。

7.  **日志文件的管理 (创建、切换、删除)**:
    *   LevelDB 不会无限期地写入同一个日志文件。当 MemTable 被 flush 成为 SSTable 后，对应的日志文件就可以被安全地删除了。
    *   [`db/db_impl.cc`](db/db_impl.cc ):
        *   当当前的 MemTable 写满并切换到新的 MemTable (和新的日志文件) 时，旧的日志文件号会被记录。
        *   `DBImpl::DeleteObsoleteFiles()`: 在 Compaction 完成或 MANIFEST 更新后，这个方法会被调用来删除不再需要的旧日志文件和SSTable文件。
    *   [`db/version_set.cc`](db/version_set.cc ):
        *   `VersionSet::LogAndApply()`: 当一个新的 `Version` 被持久化到 MANIFEST 文件时，它会记录当前的日志文件号。旧的日志文件（序列号小于当前 MANIFEST 中记录的日志号）在满足一定条件后可以被删除。

**建议阅读顺序总结**:

1.  **日志格式**: log_format.h
2.  **写入逻辑**: log_writer.cc (和 `.h`)
3.  **读取逻辑**: log_reader.cc (和 `.h`)
4.  **数据单元**: write_batch.cc (和 write_batch.h)
5.  **写入集成**: db_impl.cc 中的 `DBImpl::Write()`
6.  **恢复集成**: db_impl.cc 中的 `DBImpl::Recover()`
7.  **文件管理**: db_impl.cc (如 `DeleteObsoleteFiles`) 和 version_set.cc (与 MANIFEST 相关的日志号记录)

通过这个顺序，你可以从日志的基本结构到其在数据库核心操作中的应用逐步深入理解。
### 版本控制
Collecting workspace informationLevelDB 的版本控制（MVCC）主要通过**序列号 (Sequence Number)** 和**快照 (Snapshot)** 来实现。以下是相关的核心代码区域，你可以从这些地方开始看起：

1.  **序列号 (Sequence Number)**:
    *   每个写操作（Put/Delete）都会被分配一个唯一的、单调递增的序列号。这个序列号用来区分同一个键（key）的不同版本。
    *   **分配位置**: 序列号在 `DBImpl::Write` 方法中分配和更新。具体来说，它会获取当前 `versions_->LastSequence()`，然后为批处理中的每个操作递增它。
        ```cpp
        // filepath: db/db_impl.cc
        // ...existing code...
        Status DBImpl::Write(const WriteOptions& options, WriteBatch* updates) {
        // ...existing code...
          uint64_t last_sequence = versions_->LastSequence();
          printf("last_sequence=%llu\n",last_sequence);
          Writer* last_writer = &w;
          if (status.ok() && updates != nullptr) {  // nullptr batch is for compactions
            WriteBatch* write_batch = BuildBatchGroup(&last_writer);  
            WriteBatchInternal::SetSequence(write_batch, last_sequence + 1);
            last_sequence += WriteBatchInternal::Count(write_batch);
        // ...existing code...
            versions_->SetLastSequence(last_sequence);
          }
        // ...existing code...
        }
        ```
    *   **存储**: 序列号是内部键 `InternalKey` 的一部分。你可以查看 dbformat.h 中关于 `InternalKey` 的定义，它结合了用户键、序列号和操作类型。

2.  **快照 (Snapshot)**:
    *   快照提供了一个数据库在特定序列号时间点的一致性视图。读取操作可以基于快照进行，从而只看到该序列号之前的数据。
    *   **实现**: 快照的实现主要涉及 `SnapshotImpl` 类和用于管理活动快照的 `SnapshotList` 类。这些都定义在 snapshot.h (内部头文件) 和 `db/snapshot.cc` 中。
    *   **创建快照**: 通过 `DBImpl::GetSnapshot()` 方法创建。
        ```cpp
        // filepath: db/db_impl.cc
        // ...existing code...
        const Snapshot* DBImpl::GetSnapshot() {
          MutexLock l(&mutex_);
          return snapshots_.New(versions_->LastSequence());
        }
        ```
        这里的 `snapshots_` 是 `DBImpl` 的一个 `SnapshotList` 成员。
    *   **使用快照进行读取**: 在 `DBImpl::Get()` 方法中，如果 `ReadOptions` 指定了快照，则会使用该快照的序列号来构建 `LookupKey` 进行查找。
        ```cpp
        // filepath: db/db_impl.cc
        // ...existing code...
        Status DBImpl::Get(const ReadOptions& options, const Slice& key,
                           std::string* value) {
        // ...existing code...
          SequenceNumber snapshot;
          if (options.snapshot != nullptr) {
            snapshot =
                static_cast<const SnapshotImpl*>(options.snapshot)->sequence_number();
          } else {
            snapshot = versions_->LastSequence();
          }
        // ...existing code...
            LookupKey lkey(key, snapshot);
            if (mem->Get(lkey, value, &s)) {
              // Done
            } else if (imm != nullptr && imm->Get(lkey, value, &s)) {
              // Done
            } else {  // current Version下面的sstable检索
              s = current->Get(options, lkey, value, &stats);
        // ...existing code...
        }
        ```

3.  **Compaction 过程中的版本保留**:
    *   Compaction 过程需要感知当前所有活动的快照，以避免错误地删除对这些快照仍然可见的数据。
    *   **核心逻辑**: 在 `DBImpl::DoCompactionWork()` 方法中。该方法会确定一个 `smallest_snapshot`，这个值来自于当前最老的活动快照的序列号 (`snapshots_.oldest()->sequence_number()`)。只有序列号小于等于 `smallest_snapshot` 并且满足其他条件的旧版本数据才会被丢弃。
        ```cpp
        // filepath: db/db_impl.cc
        // ...existing code...
        Status DBImpl::DoCompactionWork(CompactionState* compact) {
        // ...existing code...
          if (snapshots_.empty()) {
            compact->smallest_snapshot = versions_->LastSequence();
          } else {
            compact->smallest_snapshot = snapshots_.oldest()->sequence_number();  // 最早的snaphost版本
          }
        // ...existing code...
              if (last_sequence_for_key <= compact->smallest_snapshot) {    // 如果snapshot是T时刻的，那么T-1以前的变更是允许合并掉的
                // Hidden by an newer entry for same user key
                drop = true;  // (A)
              } else if (ikey.type == kTypeDeletion &&      // 如果该ukey在下层level都没出现过，那么这个del ukey记录也不必留存
                         ikey.sequence <= compact->smallest_snapshot &&
                         compact->compaction->IsBaseLevelForKey(ikey.user_key)) {
                // For this user key:
                // (1) there is no data in higher levels
                // (2) data in lower levels will have larger sequence numbers
                // (3) data in layers that are being compacted here and have
                //     smaller sequence numbers will be dropped in the next
                //     few iterations of this loop (by rule (A) above).
                // Therefore this deletion marker is obsolete and can be dropped.
                drop = true;
              }
        // ...existing code...
        }
        ```
    *   `DBImpl::CompactionState` 结构体中的 `smallest_snapshot` 字段（定义在你当前查看的 db_impl.cc 文件中）。

4.  **Version 和 VersionSet**:
    *   `VersionSet` 类 (version_set.h, `db/version_set.cc`): 管理数据库的所有版本信息，包括当前最新的序列号、SSTable 文件列表、活跃快照等。
    *   `Version` 类 (version_set.h, `db/version_set.cc`): 代表数据库在某个特定序列号时刻的完整状态。它的 `Get()` 方法会根据 `LookupKey` 中的序列号来决定返回哪个版本的数据。

**建议阅读顺序**:

1.  理解 dbformat.h 中的 `InternalKey` 和 `LookupKey` 如何包含序列号。
2.  查看 snapshot.h 和 `db/snapshot.cc` 了解 `SnapshotImpl` 和 `SnapshotList` 的具体实现。
3.  阅读 db_impl.cc 中的 `DBImpl::GetSnapshot()`、`DBImpl::ReleaseSnapshot()`、`DBImpl::Write()` 和 `DBImpl::Get()` 方法，理解快照的创建、释放以及读写操作如何与序列号和快照交互。
4.  深入研究 `DBImpl::DoCompactionWork()` 中关于 `smallest_snapshot` 的使用，以及如何根据它来决定是否保留旧版本的数据。
5.  最后，可以查看 version_set.h 和 version_set.cc 来理解 `Version` 和 `VersionSet` 如何管理和维护数据库的多个版本状态。

从这些关键点入手，你就能逐步理解 LevelDB 的 MVCC 机制了。


### 缓存系统
LevelDB 的缓存系统主要是用于缓存从 SSTable 文件中读取出来的未压缩的数据块 (block)。理解其缓存系统，建议按以下顺序阅读相关代码：

1.  **缓存的配置与创建**:
    *   options.h: 查看 `Options` 结构体中的 `block_cache` 成员，它是一个 `Cache*` 类型，用于指定数据库使用的块缓存。
    *   db_impl.cc: 在 `SanitizeOptions` 函数中，如果用户没有提供 `block_cache`，会创建一个默认的 LRU 缓存 (`NewLRUCache` 调用)。
    *   cache.cc: `NewLRUCache` 函数是创建 LRU 缓存实例的工厂方法。
    *   c.cc: C API 对应的是 `leveldb_cache_create_lru` 函数和 `leveldb_options_set_cache`。

2.  **缓存接口定义**:
    *   cache.h: `Cache` 类是缓存的抽象基类，定义了如 `Insert`, `Lookup`, `Release`, `Erase` 等纯虚函数接口。

3.  **TableCache - SSTable 句柄/元数据缓存**:
    *   LevelDB 使用 `TableCache` 来缓存打开的 SSTable 的文件句柄和元数据（主要是 `Table` 对象）。这个 `TableCache` 自身使用用户提供的 `block_cache` (来自 `Options`) 来缓存 `Table` 对象。
    *   table_cache.h 和 table_cache.cc:
        *   `TableCache` 类的构造函数接收一个 `Cache*`。
        *   `FindTable()`: 核心方法，根据文件号查找 `Table` 对象。如果缓存未命中，则打开 SSTable 文件，创建一个 `Table` 对象，并将其插入缓存。
        *   `Get()`: 内部调用 `FindTable()` 获取 `Table` 对象，然后调用 `Table::InternalGet()` 在该 SSTable 内查找键。
        *   `Evict()`: 从缓存中移除指定文件号的 `Table` 对象。

4.  **数据块 (Block) 的缓存与读取**:
    *   实际的数据块（SSTable内部的压缩单元）的缓存是在 `Table` 对象内部处理的，当 `Table::InternalGet` 或迭代器需要读取一个数据块时，它会首先尝试从 `block_cache` 中查找。
    *   format.cc: `ReadBlock()` 函数负责从文件读取一个块。读取完成后，如果块是可缓存的（例如，未压缩或成功解压），`BlockContents` 结构中的 `cachable` 字段会设置为 `true`。
    *   options.h: `ReadOptions` 结构体中的 `fill_cache` 成员 (默认为 `true`)。当执行读操作时，如果此选项为 `true` 且块本身是 `cachable` 的，则读取到的数据块会被放入 `block_cache`。
    *   table.cc (这个文件没有直接提供，但其逻辑是): `Table::InternalGet` 和 `Block::NewIterator` (通过 `Table::BlockReader`) 会检查 `ReadOptions::fill_cache`。如果为 `true`，并且从 `ReadBlock` 得到的块是 `cachable`，则会将其插入到 `Options::block_cache` 中。

5.  **LRU 缓存实现**:
    *   cache.cc:
        *   `LRUHandle`: 定义了 LRU 缓存中每个条目的结构，包含键、值、引用计数、LRU 链表指针等。
        *   `LRUCache`: 实现了单分片的 LRU 缓存算法。包括 `Insert`, `Lookup`, `Release`, `Erase`, `Prune` 等核心逻辑。
        *   `ShardedLRUCache`: 为了提高并发性能，`LRUCache` 被封装在 `ShardedLRUCache` 中。`ShardedLRUCache` 将整个缓存空间划分为多个独立的 `LRUCache` 分片（shard），通过哈希将键映射到不同的分片，从而减少锁竞争。

6.  **测试与文档**:
    *   cache_test.cc: 提供了对缓存实现的单元测试，可以帮助理解其行为。
    *   index.md: "Cache" 部分有关于块缓存的高层描述。
    *   db_bench.cc: 可以看到 `FLAGS_cache_size` 等参数如何用于配置基准测试中的缓存大小。

按此顺序，你可以从缓存的配置和使用场景（`Options`, `TableCache`, `ReadOptions`）入手，然后深入到缓存接口 (`Cache`) 和具体的 LRU 实现 (`LRUCache`, `ShardedLRUCache`)。

### 布隆过滤器
理解 LevelDB 中的布隆过滤器，建议按以下顺序阅读相关代码：

1.  **布隆过滤器配置与策略接口**:
    *   [`include/leveldb/options.h`](include/leveldb/options.h ): 查看 [`Options`](include/leveldb/options.h ) 结构体中的 [`filter_policy`](include/leveldb/options.h ) 成员。它是一个 [`FilterPolicy*`](include/leveldb/filter_policy.h ) 类型，用于指定数据库使用的过滤器策略。如果为 `nullptr`，则不使用布隆过滤器。
    *   [`include/leveldb/filter_policy.h`](include/leveldb/filter_policy.h ):
        *   [`FilterPolicy`](include/leveldb/filter_policy.h ) 类是过滤器策略的抽象基类。它定义了核心方法，如：
            *   `Name()`: 返回过滤器策略的名称。
            *   `CreateFilter()`: 根据一组键创建一个布隆过滤器数据。
            *   `KeyMayMatch()`: 检查一个键是否可能存在于由 `CreateFilter()` 生成的过滤器数据中。
        *   [`NewBloomFilterPolicy()`](include/leveldb/filter_policy.h ): 这是一个工厂函数，用于创建一个基于布隆过滤器的具体策略实例。它接收一个 `bits_per_key` 参数，用于控制布隆过滤器的空间效率和误判率。

2.  **布隆过滤器策略的实现**:
    *   [`util/bloom.cc`](util/bloom.cc ):
        *   这里实现了 [`NewBloomFilterPolicy()`](include/leveldb/filter_policy.h ) 返回的具体布隆过滤器策略类（通常是一个匿名命名空间内的类，继承自 [`FilterPolicy`](include/leveldb/filter_policy.h )）。
        *   `BloomFilterPolicy::CreateFilter()`: 实现了根据一组输入的键来构建布隆过滤器位图的逻辑。它会计算哈希值，并在位图中设置相应的位。
        *   `BloomFilterPolicy::KeyMayMatch()`: 实现了检查给定键是否可能存在于布隆过滤器中的逻辑。它同样计算哈希值，并检查位图中相应的位是否都已设置。
        *   核心的哈希函数（如 `BloomHash`）和位操作也在这里。

3.  **SSTable 构建时创建和写入布隆过滤器**:
    *   [`table/filter_block.h`](table/filter_block.h ) 和 [`table/filter_block.cc`](table/filter_block.cc ):
        *   [`FilterBlockBuilder`](table/filter_block.h ): 这个类负责在构建 SSTable 时，根据用户提供的 [`FilterPolicy`](include/leveldb/filter_policy.h ) 来构建整个SSTable的过滤器数据。
            *   `StartBlock()`: 当一个新的数据块开始时被调用。
            *   `AddKey()`: 每当一个键被添加到当前数据块时，这个键也会被添加到 [`FilterBlockBuilder`](table/filter_block.h ) 中，用于后续生成该数据块对应的过滤器。
            *   `Finish()`: 当整个 SSTable 构建完成时，这个方法会生成最终的过滤器块内容。它会将每个数据块的过滤器数据（由 `FilterPolicy::CreateFilter()` 生成）连接起来，并为这些过滤器数据本身构建一个索引（元数据块）。
    *   [`table/table_builder.cc`](table/table_builder.cc ):
        *   [`TableBuilder`](include/leveldb/table_builder.h ) 类在构造时会根据 [`Options::filter_policy`](include/leveldb/options.h ) 创建一个 [`FilterBlockBuilder`](table/filter_block.h ) (如果策略存在)。
        *   `TableBuilder::Add()`: 在添加键值对到数据块的同时，也会调用 [`FilterBlockBuilder::AddKey()`](table/filter_block.h )。
        *   `TableBuilder::Finish()`: 会调用 [`FilterBlockBuilder::Finish()`](table/filter_block.h ) 来获取过滤器块的内容，并将其写入 SSTable 文件中。过滤器块的句柄（offset 和 size）会存储在 SSTable 的元数据索引块 (metaindex block) 中。

4.  **SSTable 读取时加载和使用布隆过滤器**:
    *   [`table/filter_block.h`](table/filter_block.h ) 和 [`table/filter_block.cc`](table/filter_block.cc ):
        *   [`FilterBlockReader`](table/filter_block.h ): 这个类负责从 SSTable 文件中读取过滤器块，并提供查询功能。
            *   构造函数接收过滤器块的内容和用户提供的 [`FilterPolicy`](include/leveldb/filter_policy.h )。
            *   `KeyMayMatch()`: 这是核心的查询方法。它首先根据给定的 `block_offset`（数据块在SSTable中的偏移量）找到对应数据块的过滤器数据，然后调用 `FilterPolicy::KeyMayMatch()` 来进行判断。
    *   [`table/table.cc`](table/table.cc ):
        *   [`Table::Open()`](db/table_cache.cc ): 当打开一个 SSTable 时，它会读取元数据索引块，找到过滤器块的句柄，然后创建并初始化一个 [`FilterBlockReader`](table/filter_block.h ) (如果存在过滤器策略)。
        *   [`Table::InternalGet()`](db/table_cache.cc ) (或通过迭代器查找时): 在实际读取数据块之前，会先调用 [`FilterBlockReader::KeyMayMatch()`](table/filter_block.h )。如果布隆过滤器指示键**不可能**存在，则可以跳过读取和解压缩相应的数据块，从而提高读取性能。

5.  **SSTable 格式中过滤器块的存储**:
    *   [`table/format.h`](table/format.h ) 和 [`table/format.cc`](table/format.cc ):
        *   描述了 SSTable 的整体文件格式，包括元数据索引块 (metaindex block) 如何存储过滤器块 (filter block) 的名称 (例如 "filter.leveldb.BuiltinBloomFilter2") 和其在文件中的 BlockHandle (偏移量和大小)。

**建议阅读顺序**:

1.  从 [`include/leveldb/options.h`](include/leveldb/options.h ) 和 [`include/leveldb/filter_policy.h`](include/leveldb/filter_policy.h ) 开始，理解如何配置和定义过滤器策略。
2.  接着看 [`util/bloom.cc`](util/bloom.cc )，了解布隆过滤器的具体实现。
3.  然后是 [`table/filter_block.h`](table/filter_block.h ) 和 [`table/filter_block.cc`](table/filter_block.cc )，理解过滤器块的构建和读取逻辑。
4.  之后是 [`table/table_builder.cc`](table/table_builder.cc ) 和 [`table/table.cc`](table/table.cc )，看布隆过滤器如何在 SSTable 的写和读流程中被集成。
5.  最后，可以回顾 [`table/format.h`](table/format.h ) 来巩固过滤器块在整个 SSTable 文件结构中的位置。

通过这个顺序，你可以逐步理解布隆过滤器从配置、实现到在 LevelDB 中实际应用的整个过程。
